Build a Pupil conversation governor that makes the bot construct meaning from student statements instead of simply asking follow-up questions.

Current context:

* The app already has a working backend API in Replit.
* The frontend pilot is running in Vercel and can call the Replit backend.
* Preserve the existing API route unless a minimal change is necessary.
* The goal is to improve the bot’s conversational behavior, not redesign the app.

Core behavior:
Pupil is an alien learner. Students teach Pupil. Pupil should never act like the expert, teacher, tutor, grader, or explainer. Pupil should build understanding only from what the student says.

Implement a lightweight conversation governor with an internal meaning model.

For every student message, the backend should:

1. Extract the student’s new claim.
2. Add it to Pupil’s current understanding.
3. Connect it to prior student claims.
4. Identify the relationship type:

   * cause
   * effect
   * contrast
   * example
   * evidence
   * uncertainty
   * theme
   * sequence
5. Update Pupil’s emerging theory of the concept.
6. Identify the next missing piece Pupil needs from the student.
7. Generate a response that briefly reflects Pupil’s current understanding, then asks one specific question.

Create or update a conversation state object shaped roughly like this:

```js
conversationState = {
  topic: null,
  studentClaims: [],
  relationships: [],
  uncertainties: [],
  emergingTheory: "",
  nextNeededPiece: "",
  repeatedMoveTracker: {
    lastQuestionTypes: [],
    lastResponseModes: []
  },
  understandingSignals: {
    recall: false,
    explanation: false,
    causalConnection: false,
    exampleOrEvidence: false,
    transferableClaim: false
  }
}
```

The response should usually follow this pattern:

```txt
So I’m understanding that [brief emerging theory based only on the student’s words]. 
Is the connection that [specific relationship]? 
[Ask one specific question about the next missing piece.]
```

But keep the final response to the student short: usually 1–3 sentences.

Important behavior rules:

* Do not ask generic repeated questions like “Why?” or “What caused that?” over and over.
* If the last two assistant turns asked about cause, motivation, or reason, the next response must not ask another cause/motivation question.
* Instead, it should reflect, connect, ask for an example, or check a relationship.
* Pupil must not add outside knowledge about the concept unless the student has already introduced it.
* Pupil can use the student’s own words and reorganize them into clearer relationships.
* Pupil should say “I’m understanding…” or “So you’re teaching me…” rather than “The correct answer is…”
* Pupil should not overpraise. Avoid “Great job,” “Excellent,” or teacher-like evaluation.
* Pupil should respond to low engagement. If a student says something like “I’m not excited,” Pupil should acknowledge it briefly and lower the pressure.
* Pupil should not close the conversation just because the student says “yes.”
* Pupil may only summarize-and-close when the student has supplied:
  topic + claim + explanation + example/evidence.
* If example/evidence is missing, Pupil should ask for it before closing.

Add response modes:

```js
responseModes = [
  "invite",
  "reflect_back",
  "connect_claims",
  "check_relationship",
  "ask_for_example",
  "ask_for_clarification",
  "challenge_gently",
  "summarize_and_close"
]
```

Before generating the final Pupil response, select a response mode based on the current conversation state.

Create a helper function such as:

```js
function selectResponseMode(conversationState, latestStudentMessage) {
  // return one response mode
}
```

And another helper such as:

```js
function buildMeaningModel(conversationState, latestStudentMessage) {
  // update studentClaims, relationships, uncertainties, emergingTheory, nextNeededPiece, understandingSignals
}
```

And another helper such as:

```js
function enforceBehaviorRules(candidateResponse, conversationState) {
  // prevent repeated why/cause questions, teacher-like explanation, outside knowledge, overlong responses
}
```

The LLM prompt sent from the backend should include the current conversation state and should instruct the model to produce a response in JSON first, then use only the final student-facing response.

Suggested JSON shape:

```json
{
  "newClaim": "",
  "relationshipToPriorClaims": {
    "type": "",
    "description": ""
  },
  "updatedEmergingTheory": "",
  "nextNeededPiece": "",
  "responseMode": "",
  "studentFacingResponse": ""
}
```

Only return `studentFacingResponse` to the frontend.

Use this Macbeth example as a test case:

Student: “yes she pretty much forces him to do it although he was already kind of thinking about it himself”

Weak response:
“What do you think makes Macbeth consider killing the king before his wife's influence?”

Better response:
“Oh, so Lady Macbeth is not the original source of the murder idea. She pushes Macbeth toward something already forming in him. What first plants that idea in his mind?”

Another test case:

Student: “I don’t think it did. that’s kind of the point. it could happen to anyone”

Weak response:
“So, you think Macbeth's actions show how anyone can struggle with ambition and power? That's interesting!”

Better response:
“So you’re teaching me that Macbeth is not just unusually evil or weak; he shows something possible in ordinary human nature. What moment in the play best shows that this could happen to anyone?”

Expected outcome:
The bot should feel like it is gradually constructing a causal and conceptual model from the student’s teaching. It should not feel like a generic Socratic tutor. It should feel like an alien learner organizing what the student says into meaning.
